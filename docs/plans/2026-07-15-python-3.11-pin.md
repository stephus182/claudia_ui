# Python 3.11 Revert — Remove 3.14 Compat Patches Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Delete every Python-3.14-specific monkeypatch from `claudia/app.py` and revert `.venv` to Python 3.11 — the version `pyproject.toml`, `ruff`, and `ibkr_core_mcp`'s own classifiers already declare as designed-for — with guardrails so the interpreter can never silently drift to an unsupported version again.

**Architecture:** No new features, no code that depends on Python version. Remove ~200 lines of private-API monkeypatches from `claudia/app.py` (`anyio._task_states`, `anyio.CancelScope.__exit__`, `anyio.AsyncIOTaskInfo.__init__`, `asyncio.wait_for`, `sniffio.current_async_library`, `anyio.to_thread.run_sync`), rebuild `.venv` on 3.11, add a `requires-python` ceiling plus a `.python-version` file, and verify with a real `chainlit run claudia/app.py` boot test (not just `chainlit hello`) plus the full test suite. `ibkr_core_mcp`'s already-fixed, already-tested URL-doubling bug (unrelated to 3.14, found during this investigation) gets committed as part of cleanup.

**Tech Stack:** Python 3.11 (Homebrew), pip, pytest, Chainlit CLI, git.

---

## Why (full accurate history, 2026-07-15)

`.venv` was accidentally built on Python 3.14 on 2026-06-09 because `python3.11` was never installed via Homebrew on this machine. Six monkeypatches were added to `claudia/app.py` incrementally over three separate work sessions to work around Python 3.14's stricter `asyncio.timeout().__aenter__()` check (it now raises `RuntimeError: Timeout should be used inside a task` whenever `asyncio.current_task()` is `None` — a regression against an assumption baked into `anyio`/`uvicorn`/`engineio` before 3.14): `anyio.to_thread.run_sync` and `asyncio.wait_for` on 2026-06-09 (commits `0f55a5b`, `59a7bd9`), `sniffio.current_async_library`, `anyio._task_states`, and `CancelScope.__exit__` on 2026-06-10 (commits `581aa61`, `c8e6c58`, `e6634c5`), and `AsyncIOTaskInfo.__init__` on 2026-06-30 (commit `5328146`, for the TradingView sidecar's MCP stdio loop specifically). A separate, non-monkeypatch workaround — routing custom static assets (`custom.css`/`custom.js`/the logo) through hand-rolled routes instead of Chainlit's `FileResponse`-based handler — was also added 2026-06-09 (commits `adc01ff`, `510b8f5`) for the same underlying `anyio.to_thread` bug; this one doesn't touch library internals and isn't part of the patch-removal scope below, but its justifying comment is now stale post-revert (see Task 8).

- The patches were never validated beyond boot — order staging, GDrive sync, and the TradingView sidecar's MCP stdio loop were never live-tested end-to-end on 3.14.
- They reach into `anyio`'s and `asyncio`'s private internals (`_task_states`, `CancelScope.__exit__`, `AsyncIOTaskInfo.__init__`) with no compatibility guarantee — every future `anyio`/`uvicorn` upgrade is a fresh chance to silently re-break something "fixed" here, on a codebase that stages real trades.
- A claimed 4th occurrence (`IBKRWebSocket.connect()` hitting the identical bug, blocking live P&L/execution tracking) did not reproduce under a test of `IBKRWebSocket.connect()` run as a standalone `asyncio.run()` script. That test has a real gap, though: `docs/project-status.md`'s 2026-07-10 entry on this exact symptom found the same thing — standalone, it works fine — but explicitly concluded the failure is "scoped to running inside Chainlit/uvicorn's asyncio context," a condition the standalone test (both the 07-10 one and this plan's) never actually reproduces. That investigation was flagged "still open" and never completed (the proposed diagnostic step was never added, confirmed via `git log`). So this claim was genuinely unresolved before this plan, not disproven by it — though it's moot either way once reverted to 3.11, since the whole bug class is 3.14-only. Separately: `ibkr_core_mcp`'s default `gateway_url` already includes `/v1/api`, and `streaming.py`'s `IBKRWebSocket.__init__` was appending `/v1/api/ws` on top of it, producing `/v1/api/v1/api/ws` on every connection attempt — a real, independent, version-agnostic bug, already found, fixed, and tested (see Task 1), which may also explain some or all of the observed `ExecutionListener` failures regardless of the asyncio question above.

Decision: stop patching around Python 3.14 entirely and revert to 3.11. Full functionality and robustness are the explicit priority; chasing individually-discovered occurrences of an unmapped ecosystem-wide regression is the wrong shape of engineering effort for an order-execution codebase. A UI-framework migration away from Chainlit (to NiceGUI, discussed separately) remains a deliberate future initiative and is explicitly out of scope here — this plan only removes the patches and restores the originally-designed-for interpreter.
