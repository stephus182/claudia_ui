# Python 3.11 Revert — Remove 3.14 Compat Patches Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Delete every Python-3.14-specific monkeypatch from `claudia/app.py` and revert `.venv` to Python 3.11 — the version `pyproject.toml`, `ruff`, and `ibkr_core_mcp`'s own classifiers already declare as designed-for — with guardrails so the interpreter can never silently drift to an unsupported version again.

**Architecture:** No new features, no code that depends on Python version. Remove ~200 lines of private-API monkeypatches from `claudia/app.py` (`anyio._task_states`, `anyio.CancelScope.__exit__`, `anyio.AsyncIOTaskInfo.__init__`, `asyncio.wait_for`, `sniffio.current_async_library`, `anyio.to_thread.run_sync`), rebuild `.venv` on 3.11, add a `requires-python` ceiling plus a `.python-version` file, and verify with a real `chainlit run claudia/app.py` boot test (not just `chainlit hello`) plus the full test suite. `ibkr_core_mcp`'s already-fixed, already-tested URL-doubling bug (unrelated to 3.14, found during this investigation) gets committed as part of cleanup.

**Tech Stack:** Python 3.11 (Homebrew), pip, pytest, Chainlit CLI, git.

---

## Why (full accurate history, 2026-07-15)

`.venv` was accidentally built on Python 3.14 on 2026-06-09 because `python3.11` was never installed via Homebrew on this machine. That same day, three monkeypatches were added to `claudia/app.py` (commit `59a7bd90`) to work around Python 3.14's stricter `asyncio.timeout().__aenter__()` check (it now raises `RuntimeError: Timeout should be used inside a task` whenever `asyncio.current_task()` is `None` — a regression against an assumption baked into `anyio`/`uvicorn`/`engineio` before 3.14). This got Chainlit's boot-level Socket.IO/ASGI path working, but:

- The patches were never validated beyond boot — order staging, GDrive sync, and the TradingView sidecar's MCP stdio loop were never live-tested on 3.14.
- They reach into `anyio`'s and `asyncio`'s private internals (`_task_states`, `CancelScope.__exit__`, `AsyncIOTaskInfo.__init__`) with no compatibility guarantee — every future `anyio`/`uvicorn` upgrade is a fresh chance to silently re-break something "fixed" here, on a codebase that stages real trades.
- A claimed 4th occurrence (`IBKRWebSocket.connect()` hitting the identical bug, blocking live P&L/execution tracking) did **not** reproduce under a faithful test of the actual invocation path (`asyncio.create_task()`-wrapped, matching how `ExecutionListener.start()` really calls it, against both the real `websockets` 16.0 library and a local self-signed-TLS test server). The real, empirically-observed `ExecutionListener` failures are much better explained by an unrelated bug: `ibkr_core_mcp`'s default `gateway_url` already includes `/v1/api`, and `streaming.py`'s `IBKRWebSocket.__init__` appended `/v1/api/ws` on top of it, producing `/v1/api/v1/api/ws` on every connection attempt. That bug has already been found, fixed, and tested independently of this plan (see Task 1) — it affects every Python version, not just 3.14.

Decision: stop patching around Python 3.14 entirely and revert to 3.11. Full functionality and robustness are the explicit priority; chasing individually-discovered occurrences of an unmapped ecosystem-wide regression is the wrong shape of engineering effort for an order-execution codebase. A UI-framework migration away from Chainlit (to NiceGUI, discussed separately) remains a deliberate future initiative and is explicitly out of scope here — this plan only removes the patches and restores the originally-designed-for interpreter.
